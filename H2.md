## 3. Next Feature: Win Condition (🟡 → ✅)

**Chosen because:** highest readiness (counter already exists), it's the semantic unblock for mistake allowance, and it's the *first real exercise of the `initAll()` replay path* — the game currently loops forever and has never re-inited, so the win condition is how the init work gets validated rather than assumed.

### Live scaffolding already present
- `currRound` increments in `startRound()` (`RoundManager:91`) but is **never read** — no win check anywhere.
- `RoundManager.init()` re-seeds `currRound = 0` — replay path exists but untested.
- The round loop re-arms in **two** places, both of which a terminal state must intercept:
  - `RoundManager.onCorrectAnswer` → `onNextRound()` → `startRound()` (the success path).
  - `InputHandler.handleTimerEnd` **timeout-with-<3-selected** branch: after the overtime tween it sets `roundActive = true` + `canClick = true` and lets play continue (`InputHandler:121–130`). Currently there is **no game-over on timeout** — the spec says timer-to-zero is game over, but live code just resumes.

### The architectural fork to resolve BEFORE writing code
**Where does the win check belong?** Three candidates, each with a different consequence for whether the *final* cleared round renders before the win screen appears:

| Placement | Consequence |
| :-- | :-- |
| Inside `startRound` (before `currRound += 1`) | Checks on *entry* to the next round → the winning round's target/board may not have rendered the "you won on this one" beat. |
| Inside `onCorrectAnswer` (after score applied, before `onNextRound`) | Checks at the moment of the clearing answer → cleanest "win fires on the correct submission that crosses the threshold." Most likely correct. |
| In the `onNextRound` callback (`LocalScript`) | Keeps `RoundManager` unaware of terminal logic; composition root decides whether to start another round or end. Cleanest boundary, but splits win logic across two files. |

This is the first thing to pin next session. **Do not pre-decide it for the user** — surface the trade-off, let them make the executive call.

### Secondary decisions this feature forces
- **Terminal display state.** Needs a new `winData` / `gameOverData` state in `upperDisplay` (and maybe `lowerDisplay`). This is the **first extensibility test** of the refactored state-table pattern — if adding it is a ~2-line addition per state, the refactor proved its worth. Watch whether the `score`-priority guard (`currState == "score"` blocks non-score writes) interferes with painting a terminal state — a win firing right after `onCorrectAnswer` sets `score` state may be *blocked by its own priority guard*. **This is a real interaction to trace**, not hypothetical.
- **Timeout = game over.** The `handleTimerEnd` <3-selected branch currently resumes play; the spec wants game-over. Win-condition work is the natural place to wire the losing terminal state too.
- **Two round targets: 5 and 6.** Win condition is entangled with mistake allowance from the start (1 mistake → target becomes 6). Build the 5-round check such that the *target* is a variable, not a literal, so mistake allowance later flips 5→6 without restructuring.

## 4. Feature After That: Mistake Allowance (📋, High Value)

Depends on win condition (raising the requirement to 6 is meaningless without a requirement). Live readiness:
- **Detection already exists.** `AnsChecker.isCorrect` returns false for a valid straight line that misses the target; `isValid` returns a *string* reason for structural failures. So "valid line, wrong sum" (a genuine mistake) is already distinguishable from "invalid layout" (no penalty) — see `InputHandler.handleAnsCheck:43`, which currently collapses both into one `incorrect` tween.
- **Needs:** a mistake counter in `RoundManager` (+ `init` seed), the 5→6 round-target flip, and the spec's mistake-compensation (15→20s round timer, fewer board mods). Reveal + blacklist are separately 📋 and can defer.
- **Gotcha to flag when we get there:** `handleAnsCheck` doesn't currently branch on *why* an answer failed. Consuming the mistake budget requires splitting the `validity ~= true or correctness == false` condition into its two cases.

## 5. Standing Architectural Notes (live-verified this session)

- **Font-colour ownership open question (parked).** Split three ways per Doc 4 — does memorization font colour move into `HexAnimator` (which owns the part tween) as a proper tween, or stay a `RoundManager` property write? Same state/presentation seam as the old gating question. Still unresolved.
- **`modifyUpperState`'s `pairs` loop widens the contract.** `for i,v in pairs(contextArgs) do upperDisplay[state.."Data"][i] = v end` copies *any* passed key into the state table — a caller typo (`roundeTime`) creates a junk key silently while the real field stays stale. Elegant at n=2 fields, but trades an explicit contract for an implicit one; flagged against CLAUDE.md's "expressive over clever." Not urgent; revisit if a third field joins.
- **`scoreData` is not self-contained.** The `score` state writes the score value into `defaultData.score` and only `scoreGain` into `scoreData`, so `scoreData` can't render without `defaultData` primed. Intentional (score is an overlay on default), but worth remembering when adding terminal states.
