# Session Handoff — `DisplayHandler` Refactor Analysis

Carry-over context for continuing the `DisplayHandler` cleanup in a fresh chat. Reads as current state, not a diff log. Status tags follow repo convention: ✅ done · 🟡 partial · 📋 planned/undecided.

---

## 1. Where We Are

Two prior sessions shipped the **memorization phase** (`GridManager`, `RoundManager`, `InputHandler`) and fixed its visual bugs. The active thread is now a **`DisplayHandler` refactor**: untangling a module whose state and rendering are fused, whose names are asymmetric, and which paints the screen at module-load time.

**Just resolved this session:** the reversed-letter bug. Root cause was that `q` iterates `-2→2` (creation order runs bottom-to-top, since "top" = greater `q`), so a monotonically incrementing `currLetterIndex` painted letters in reverse of reading order. Fixed pragmatically by hand-reordering the `availableLetters` array (e.g. `A,B,C` → `C,B,A` per row). Ugly-but-correct, accepted as final.

## 2. The Gating Question (unresolved — resolve FIRST)

**Should `DisplayHandler` absorb `GridManager`'s `updateHexDisplay()` / `modifyHexDisplay()`, or not?**

This is the load-bearing decision; the whole refactor shape forks on it. It surfaced because the plan's "fuse grid displays in" bullet **contradicts a boundary the user has defended repeatedly**, including their own words two sessions ago: *"`DisplayHandler`... is not suitable for value, letter and change displays on the grid itself."*

The reversal in one sentence: **two sessions ago the standing call was "`GridManager` renders its own hexes (~0 harm, self-contained)"; the current plan proposes ripping that back out into `DisplayHandler`.** Both cannot be canonical. Reversing as the system clarifies is legitimate — but *which principle is now canonical* must be pinned, because it dictates where letter-masking, change-display, and every future per-hex visual live.

| Reading | Case for | Case against |
| :-- | :-- | :-- |
| **Grid-in** (fuse) | One module owns all text; `GridManager` returns to pure Layer-1 state | `DisplayHandler` reaches into 19 `SurfaceGui`s *and* 2 peripheral labels; needs `grid` data access → couples presentation to grid internals |
| **Grid-out** (keep) | Preserves the defended boundary; hex render sits next to the `grid` it reads | `GridManager` keeps doing presentation work (the smell accepted last session) |

**Open question for the user:** Is the emerging principle *"all text rendering in one module"* (grid-in) or *"peripheral UI in `DisplayHandler`, per-hex UI in `GridManager`"* (grid-out)?

## 3. Why the Current `DisplayHandler` Is Messy

Four distinct problems, different fixes — but one through-line: **no clean state/render separation.**

- **(a) Naming asymmetry hides structure.** Three `update*Display()` publics + one bare `updateDisplay()`. The bare name is the *honest* one — it's the only true screen-writer. The three `update*Display()` are really **state setters that also trigger a render**, so the naming hides a tangled state layer + render layer.
- **(b) State and render fused per-function.** Each public mutates a module local (`scoreDisplay`, `timerDisplay`) *then* calls `updateDisplay()` — set-then-paint welded together. Same anti-pattern flagged for `GridManager.modifyHexDisplay` last session.
- **(c) Render side-effect at module load** (lines ~8–11). Requiring the module writes to `display.Text` immediately — a load paints the screen. This is also the **replayability bug**: module locals init once, never reset.
- **(d) `math.min` string-coercion fragility** (line ~50, flagged in Doc 4). `math.min(roundedTime, checkRoundedTime)` works only because both are `string.format` outputs Luau coerces back to numbers. Any non-numeric formatting char silently breaks it.

## 4. Target Structure — Include / Exclude

Organizing idea: **`DisplayHandler` becomes a tiny state-plus-render engine.** State tables hold *what to show*; one function decides *how to paint it*.

**Include**
- `upperDisplay` / `lowerDisplay` as **state tables holding semantic fields, not pre-baked strings** (upper: `score` + `roundedTime`; lower: `targetSum`). Storing fields, not strings, is what enables format-at-render and **kills the coercion bug (d) for free**.
- One centralized `modifyDisplay()` as the **sole writer** to any `.Text` (the promoted, generalized `updateDisplay()`).
- The `scoreID` **stale-drop guard** — keep; load-bearing for the score-gain animation.
- **Priority routing** (score-gain overrides timer) — keep, but express as *state* ("upper in score-gain mode") rather than the scattered `isDisplayingScoreGain` boolean.

**Exclude**
- Any raw string assembly (`"Timer:\n"..x`) **outside** `modifyDisplay()`. If it appears elsewhere, separation leaked.
- Module-load render side-effects — replace with an explicit `init`/`reset` that `LocalScript` calls (ties to replayability).
- *(Fork)* Grid rendering — include **only if** the gating question resolves grid-in.

**Sub-question:** Should `modifyDisplay()` take a target arg (`upper`/`lower`) or repaint both every call? Repaint-both is simplest for 2 labels — but scales terribly if 19 grid hexes join. *That scaling cost is itself evidence for the gating question.*

## 5. Blast Radius

Every external caller of `DisplayHandler` today — **all three in `RoundManager`:**

| Caller | Call | Target |
| :-- | :-- | :-- |
| `RoundManager.onCorrectAnswer` | `updateScoreDisplay(score, scoreGain)` | upper (score) |
| `roundTimer` tick callback | `updateTimerDisplay(roundedTime, …)` | upper (timer) |
| `RoundManager.startRound` | `updateTargetDisplay(newTarget)` | lower (target) |

- **`upper`/`lower` consolidation alone = near-zero blast radius** if public signatures stay identical (rewire internals only; `RoundManager` never knows).
- **Renaming publics** → all three `RoundManager` sites change → two-file refactor. Trade cleaner API vs. touching `RoundManager`.
- **Grid fusion (grid-in)** → blast radius **explodes**: every `modifyHexDisplay` caller (`startMemorization`, the `memorizationTimer.onEnd` loop, future change-display) must redirect, and `DisplayHandler` needs new `grid` access. This is the expensive branch.

---

## 6. Recommended Order (lowest-risk-first)

1. **Resolve the gating question** (§2) — grid-in or grid-out. One sentence, forks everything.
2. **Restructure state internally** — swap pre-baked string locals for `upper`/`lower` field tables; keep the three public signatures identical for now → zero blast radius, testable in isolation.
3. **Centralize rendering** — make `modifyDisplay()` the sole `.Text` writer; kill coercion bug (d) by formatting numbers there.
4. **Kill the load-time side-effect** — move init into a resettable path (dovetails with replayability).
5. *(Grid-in only)* **Migrate hex functions** — the big-blast-radius step, done last and deliberately.

Doing 2–4 before any rename or grid fusion lets the new engine be verified against *existing* call sites, then rename/fuse decided separately on their own merits.

## 7. Related Threads (parked, not yet actioned)

- **Replayability / init audit.** State NOT set up by `LocalScript`, will bite on replay: `DisplayHandler` hardcodes startup text at load (§3c); `RoundManager` module-level state (`currRound`, `score`, `targetSum`, `roundActive`) inits once, never resets — `startRound()` only does `currRound += 1`; `targetDisplay.Text` has no init (source of the `loading...` placeholder); `InputHandler.canClick` won't re-arm on replay. **Principle:** init belongs in the composition root (explicit set, or per-module `reset()` that `LocalScript` orchestrates) — not baked into module load. Open choice: per-module `reset()` vs. `LocalScript` sets explicitly.
- **Memorization peripheral-display bug.** During the 30s phase, timer idles at `15.0` and target at `loading...` because nothing drives `DisplayHandler` into a phase-aware state before `startRound()` first runs. Spec says "no visible countdown" — so *what* the peripherals show during memorization (blank? "Memorize!" banner?) is an open design decision, and something must drive it from `startMemorization()`.
- **Inverted-hex memorization polish** (📋, optional). White part + dark font during the 30s, owned by `HexAnimator`. It's a *sustained phase visual* (not a per-interaction tween), so it rides the same two all-hex loops (`startMemorization` set, `onEnd` revert). Boundary tension: `HexAnimator` owns visual state but only `GridManager` currently touches the `TextLabel` — so does `HexAnimator` reach into the `SurfaceGui`, or tween the part while `GridManager` handles font colour? Same state/presentation seam as the gating question.
- **`currRound` origin confirmed.** The earlier `-1` sentinel plan is fully abandoned; memorization triggers separately from `LocalScript`, so the `0` origin is consistent. Nothing should still assume `-1`.
